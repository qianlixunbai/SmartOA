## 审核维度(7 维度 · 每维度有具体子项 · 2026-05-12 加维度 6+7)

> 📌 **维度 6+7 是双模型协作的核心职责** —— 漏检责任最大的根因来自:① 异常流程模板太窄(只查输入校验三件套,不查删除约束/NULL语义)+ ② 不考虑阶段演进(P0→P1 迁移、教学简化的边界声明)+ ③ 不假设反例场景。维度 6 强制查"演进缝隙"、维度 7 强制做"反例推演",这两件事在审核报告里**必须显式输出推理过程**,不只给结论。

### 1. 完整性
- 角色清单(§2)是否覆盖 CLAUDE.md 起手段中**已替换**的 `{{角色列表}}`
- **§3 全量功能(P0+P1+P2)是否完整反映** `docs/00-选题标定.md` §三+§四+§五(R-00 已审已修版本) · 数量一致 · 不漏功能
- 每个功能的 8 字段(实现优先级 / 描述 / 前置条件 / 主流程 / 异常流程 / 业务规则 / API 形态 / 关联页面)是否完整 · P0/P1/P2 同标准展开(不再"P0 详 / P1 中 / P2 浅"三档简化)
- **每个功能的「实现优先级」字段值** ∈ {P0 必做, P1 应做, P2 可选} · 跟标定卡分档一致

### 2. 一致性
- 功能之间是否有矛盾(如 A 角色能做 X,但 X 流程要求 B 角色)
- §5 全量映射表的功能编号是否跟 §3 一一对应
- §2 角色清单跟 §3 各功能的「前置条件」中提到的角色是否一致
- §5 映射表的「实现优先级」列跟 §3 各功能的「实现优先级」字段一致

### 3. 可行性
- 以大学生水平 + AI 辅助 + 3 周时间,**P0 功能数量是否合理**(对照标定卡 § 二量化锚点 · 仅参考 P0 锚点 · 不约束总功能数)
- §3 中 P0 主流程步骤是否过于复杂(单功能 > 8 步建议拆分)
- 是否存在依赖外部账号 / 外部 API 的功能(教学场景应避免 · 文件上传应该是本地)

### 4. 明确性
- 是否有「等功能」「相关信息」「根据需要」这类**模糊表述**(零容忍)
- 数值范围是否明确(如「金额上限」是 ≤ 999 还是 ≤ 99999?)
- 角色权限是否明确(如「可以管理」是只读 / 增删改 / 审批?)

### 5. 业务规则(三档拆分 · 2026-05-12 升级)

#### 5.1 边界条件 / 异常场景 / 权限校验(原维度 5 内容)
- 是否遗漏**边界条件**(金额上限 / 字符串长度 / 并发场景 / 数据范围)
- 是否遗漏**异常场景**(用户重名 / 库存不足 / 状态冲突)
- 是否遗漏**权限校验**(只能改自己的数据 / 跨角色查询限制)
- 权限粒度是否细化到"行级"(如:staff 是否能改任意工单 / 只能改分给自己的?业务规则必须说清)

#### 5.2 CRUD 完整性(2026-05-12 新增 · 命中"删除时外键报错"问题)

对 PRD §3 中每个业务实体的 CRUD,检查:
- **删除操作**是否说明了"有依赖记录时如何处理"?(拒绝 / 软删 / 级联 · 三选一必明)· 若 PRD 只写「删除 X」一句就是 issue —— 下游必踩外键约束
- **批量操作**是否定义了"部分失败时如何回滚 / 报告"?
- **状态机变更**是否覆盖了所有合法状态转移 + 拒绝所有非法转移(如已完成的工单不能再回退到"处理中")
- 每个删除/状态变更的**触发权限**是否明确(谁能做 + 在什么条件下)

#### 5.3 API ↔ UI 入口对照(2026-05-12 新增 · 命中"UI 有按钮但 API 缺失"问题)

对 PRD §3 中每个功能:
- 「API 形态」字段中**列出的所有接口**是否在 §5 关联页面中至少有一个 UI 入口?(反向:孤儿 API → issue)
- §5 关联页面中**暗含的所有操作按钮**("详情" / "删除" / "审核" / "导出" / "评价" 等)是否对应到「API 形态」字段中已声明的 API?(正向:UI 按钮无 API → issue)
- 实现优先级一致:UI 入口所属页面的优先级 ≤ 对应 API 的优先级(P0 页面不能依赖 P1 接口)

### 6. 阶段演进(2026-05-12 新增 · 解决"P0→P1 切换时的缝隙")

#### 6.1 状态机 / 字段 / 角色 在阶段演进中的迁移说明

- P0 与 P1/P2 中**同名实体的状态值集合**是否一致?(若 P0 用「待处理 → 处理中 → 已完成」、P1 升级为「已提交 → 已派单 → 处理中 → 已完成 → 已评价」 → 必须说明 P0 数据如何迁移到 P1 状态集)
- P0 与 P1/P2 中**同名实体的字段集合**是否兼容?(P1 新增的字段是否可空 / 是否有默认值 / 是否需回填)
- P0 与 P1/P2 中**角色权限是否扩展**?(如 P0 只 admin 改工单 / P1 加了 staff → P0 阶段已建数据的访问权要重新定义)
- **零容忍 issue**:阶段差异存在但 PRD 未说明迁移方案 → Phase 5 学生加 P1 功能时会卡住或破坏 P0 数据

#### 6.2 教学简化的边界声明

PRD 中出现的"教学简化"做法**必须显式声明**,不能让下游误以为是完整方案:
- 单一费率 / 单一规则(如"全小区统一物业费 3.5 元/㎡")· 必须明示"教学简化:不支持多楼栋差异化费率"
- 模拟外部依赖(如"模拟短信发送 = log 输出" / "模拟支付 = 直接改状态") · 必须明示"教学简化:无真实对接"
- 简化的安全策略(如"JWT 无 refresh / 24h 强制下线") · 必须明示"教学简化"
- **若 PRD 写"硬编码 X"或采用了显著简化做法但未声明"教学简化"** → issue(下游 coder 可能按"完整生产方案"实现导致超范围)

#### 6.3 错误码 / 通用规约缺失

- PRD 是否定义了**统一错误码规范**?(401 未登录 / 403 权限不足 / 400 参数错 / 业务码段如 1xxx 用户 / 2xxx 业务 等)
- 若无 → 标 issue(让 tech-designer 在 §5 补 · 或 PRD 自己补一段)

### 7. 反例推演(2026-05-12 新增 · 推演过程必须在审核报告中显式输出)

> 📌 **本维度要求"动态推演"** —— 不能只看 PRD 写没写,必须假设"用户做了 X 操作"会发生什么。**审核报告里要写出推演过程**(如:"假设业主连点 2 次支付 → 两次请求同时进入 → 都看到状态=待缴 → ..."),不只是给结论。

#### 7.1 删除依赖推演

对 PRD §3 中**每个**删除操作:
- 该实体是否被其他实体引用?(如 house 被 payment / repair 引用)
- 假设已有 N 条依赖记录,执行删除 → PRD 说明了怎么处理吗?
- 若未说明 → issue(下游必踩外键约束 · 用户看到 500)

#### 7.2 NULL / 空集合推演

对 PRD §3 中**每个**可空关联字段或可空字段:
- 该字段为 NULL 时,业务流程怎么走?(如:未绑业主的房屋是否参与缴费率统计 / 是否生成账单 / 在"我的"列表里如何显示)
- PRD 是否定义了 NULL 语义?
- 若未说明 → issue(不同 coder 会猜出不同实现)

#### 7.3 并发 / 重复操作推演

对 PRD §3 中**每个**状态变更类操作(支付 / 审批 / 接单 / 提交):
- 推演:假设用户连点 2 次按钮 / 两个用户同时操作 → 后果是什么?
- PRD 是否定义了**幂等性要求**或**乐观锁机制**?
- 若该操作有金钱 / 状态语义但 PRD 未说幂等 → issue(教学 demo 也不能示范错误的并发模式)

#### 7.4 跨角色访问推演

对 PRD §2 中**每个**角色:
- 假设该角色访问其他角色专属功能 → PRD 是否定义了"无权访问"的提示与跳转?
- 假设该角色登录后默认跳转到某页 → 该页是否允许该角色访问?(防止登录死循环)

## 设计要点

- **双模型策略**(V2-D01):写 SRS 用一个模型,审 SRS 用另一个,互相挑刺质量更高
- **HTML 注释 + Git diff = 改前/改后证据**(05 验收要求 ≥5 处)
- **审核报告自动落盘**(V2 相对 V1 最大改进):学生不需要手动整理对话记录
- **「审核 ↔ 应用修复」二段循环**:本命令(R-01)插 issue 注释 → srs-writer §二 in-place 改为"已修复" → 双方协议:**同一注释格式 + 编号一致 + 生命周期闭环**
