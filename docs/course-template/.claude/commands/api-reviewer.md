## 审核维度(7 维度 · 每维度有具体子项 · 2026-05-12 加维度 6+7)

> 📌 **维度 6+7 是双模型协作的核心职责** —— 漏检责任最大的根因来自:① 单文档审 + ② 清单式静态检查 + ③ 不审反例场景(资源不存在 / 越权 / 幂等 / 边界值)。维度 6 强制做"跨文档对账"、维度 7 强制做"反例推演",这两件事在审核报告里**必须显式输出推理过程**,不只给结论。

### 1. RESTful 规范性
- HTTP 方法是否合理(GET 列表/详情 · POST 创建 · PUT 更新 · DELETE 删除 · **禁止用 GET 改数据**)
- URL 是否资源化(`/api/users/{id}` 而非 `/api/getUserById?id=`)
- 状态码使用是否合理(2xx 成功 / 4xx 客户端错 / 5xx 服务端错)
- 路径参数是否用 `{}`(不是 query 参数模拟)
- 资源用复数名词(`/api/users` 而非 `/api/user`)

### 2. 完整性
- 每接口 5 项必填字段是否齐全(功能 / 鉴权 / 请求参数表 / 成功响应 JSON / 异常响应表 · 对照 api-designer §3 自检)
- 分页接口是否含 `pageNum` + `pageSize` 参数
- 复杂请求是否给了 JSON body 例子
- §4 全局异常码 5 项是否齐全(400/401/403/404/500)
- §4 业务异常码是否按模块编号(1xxx-9xxx · 每模块预留 100 个 code)

### 3. 一致性
- 统一前缀 `/api`(所有接口必含)
- 统一响应格式 `Result<T>`(`{code, message, data}` · 引用 CLAUDE.md §一·三)
- 命名风格一致(参数 camelCase · 不要混用 snake_case)
- 时间字段格式一致(ISO 8601 序列化 · LocalDateTime → `2026-05-10T08:30:00`)
- 错误 code 跟 §4 异常码表一致(不要在 §3 自创 code)

### 4. 安全性
- 哪些接口需 JWT(对照 PRD §3 全量功能 · 登录后操作必含)
- 哪些接口需角色权限校验(管理员 vs 普通用户 vs 全角色)
- 是否有越权风险(用户能改/删别人的数据吗?接口是否含 owner 校验?)
- 密码字段是否在响应中过滤(不要返回明文/哈希)
- 敏感操作(改密码/删除/转账)是否有二次确认或审计

### 5. 业务覆盖
- 与 `docs/PRD.md` §3 **全量功能(P0+P1+P2)**是否一一对应(每个功能至少有 1 个对应接口)
- 每个接口的「实现优先级」标签(§2 接口清单优先级列)是否跟 PRD §3 各功能的优先级一致
- **跨档依赖检查**:有无 P0 接口依赖 P1/P2 接口(若有 → 高严重度 issue · 提醒回 Phase 0 跑 R-00 应用修复)
- 是否漏关键接口(列表分页 / 详情 / CRUD 完整性)
- 与数据库表结构是否对应(每个表至少有"列表 + 详情 + 创建"3 接口 · 否则 entity 用不上)
- **(参考 · 不再硬约束)** 接口数偏离 `docs/00-选题标定.md § 二` P0 锚点(通常约 10)的部分若 ≥ 50% 来自 P1/P2 全量驱动属正常 · 不算 issue

### 6. 跨文档对账(强制 4 类逐项对账 · 2026-05-12 新增 · 漏检责任最大的硬职责)

> ⚠️ **本维度零容忍漏检**:不做对账等于审核没做。任一对账失败均标 🔴 **高严重度** issue。
>
> 📌 **对账操作规约**:① 提取 PRD §3 「API 形态」+ DATABASE §2 表字段 + TECH §3 UI 按钮 + TECH §5 错误码段 作为"参考集" ② 扫 API_DESIGN 各处"被引用项",逐一在参考集中查 ③ 任何"找不到出处"或"反向出处缺失"都是 issue ④ 对账结果必须在审核报告中显式列出(参考集 + 被检集 + 差集)。

#### 6.1 接口 ↔ PRD「API 形态」对账

- 列出 PRD §3 各功能「API 形态」字段中声明的全部接口(HTTP 动词 + URL · 参考集)
- 列出 API_DESIGN §2 接口清单的全部接口(被检集)
- **零容忍 issue · 正向**:PRD 声明但 API_DESIGN 缺的接口 → 实现阶段功能跑不通
- **零容忍 issue · 反向**:API_DESIGN 有但 PRD 找不到出处的接口 → 死代码 / 超 PRD 范围

#### 6.2 接口 ↔ DATABASE 字段对账

对 API_DESIGN §3 接口详情中的"请求参数表 + 响应 JSON":
- 列出涉及的所有业务字段(参考集)
- 列出 DATABASE_DESIGN 各表的全部字段(被检集)
- **零容忍 issue**:接口字段在 DATABASE 找不到对应列 → 下游 service-coder 字段无源
- **字段类型一致性**:API 声明的类型(Long / String / BigDecimal / LocalDateTime)必须跟 DATABASE 列类型(BIGINT / VARCHAR / DECIMAL / DATETIME)一致

#### 6.3 接口 ↔ TECH §3 UI 按钮对账

- 列出 TECH §3 路由表「备注」+ §6 页面原型提到的可交互元素("详情" / "删除" / "编辑" / "导出" / "评价" 等)作为参考集
- 列出 API_DESIGN §2 接口清单(被检集)
- **正向**:UI 按钮 → 必有对应 API
- **反向**:已声明 API → 必有 UI 入口或调用方(防孤儿 API)
- 实现优先级一致:UI 按钮所属页面优先级 ≤ 对应 API 优先级

#### 6.4 错误码段 ↔ TECH §5 错误码规范对账

- TECH §5 规范的业务码段(如 1xxx 用户 / 2xxx 业务 / 3xxx 数据)
- API_DESIGN §4 业务异常码表实际使用的 code 值
- **零容忍 issue**:API 使用的 code 落在 TECH 未定义的段(如 9999 / 8888 自创)
- **零容忍 issue · 反向**:同一类业务错误在不同接口中用了不同 code 值(命名不一致)

### 7. 反例推演(强制 4 个反例 · 2026-05-12 新增 · 推演过程必须在审核报告中显式输出)

> 📌 **本维度要求"动态推演"** —— 不能只看 API 文档字面写没写,必须假设"客户端发起 X 请求"会怎样。**审核报告里要写出推演过程**(如:"假设客户端 GET /api/users/99999 但 id=99999 不存在 → 后端 Service 返回 null → Controller 怎么响应?返回 200 + null?还是 404?还是 Result.error(2001)?——若 API_DESIGN 未明示就是 issue"),不只是给结论。

#### 7.1 资源不存在推演

对每个含路径参数 `{id}` 的接口(GET / PUT / DELETE / 子资源):
- 当客户端传入"不存在的 id"时,API 应该返回什么?(404 状态码?200 + null?Result.error(业务码)?)
- API_DESIGN 是否在"异常响应表"中明示该场景?
- **issue**:含 `{id}` 但异常响应表无"资源不存在"行 → 不同 coder 会写不同实现 · 前端处理混乱

#### 7.2 越权推演(行级权限)

对所有"读 / 改 / 删 / 操作他人数据"的接口:
- 角色 A 携 token 调用本属于角色 B 的接口 → 返回什么?(403 + 1003 权限不足?)
- **同角色行级权限**:同是 owner 角色,A 用户能否查 / 改 / 删 B 用户的数据?接口是否做了 `WHERE owner_id = currentUserId` 隐式约束?
- API_DESIGN 是否明示"鉴权"字段(JWT 必需 / 角色限制) + "行级权限"?
- **issue · 高**:写操作类接口无角色/行级权限说明 → 任意用户可越权改他人数据(数据安全漏洞)

#### 7.3 并发幂等推演

对状态变更类接口(支付 / 审批 / 接单 / 提交 / 创建唯一资源):
- 客户端连点 2 次按钮 / 网络抖动重发同一请求 → 后端会不会执行 2 次?(产生 2 个订单 / 双倍扣款 / 重复审批)
- API_DESIGN 是否有幂等性方案?(① 数据库唯一索引 ② Redis Token ③ 业务字段乐观锁 ④ 状态机条件 UPDATE)
- **issue**:状态变更接口无幂等性方案 → 教学 demo 也会演示错误的并发模式

#### 7.4 分页 / 边界 / 空集合推演

对所有列表类接口:
- `pageSize` 是否有最大值约束?(防止客户端传 pageSize=999999 拖垮 DB)
- 无数据时返回什么?(`Result<List<T>>` 中 data 应是 `[]` 还是 `null`?)
- 排序参数是否限制为白名单字段?(防 SQL 注入式排序)
- **issue · 中**:列表接口无 pageSize 上限 / 空集合返回 null / 排序字段无白名单
- **空字符串 / 空 body 推演**:POST 接口字段为空字符串 / null / 缺字段 → @Valid 校验返回什么?
