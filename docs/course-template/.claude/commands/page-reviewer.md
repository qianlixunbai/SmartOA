## 审核维度(5 维度 · 每维度有具体子项)

> 📌 **本命令的核心职责是"跨文档对账"** —— 维度 2/3/4 都是对账维度,**必须显式输出对账结果**(参考集 + 被检集 + 差集)· 即便结论是"对账通过"也要把证据列出。

### 1. §6 内部完整性

- §6 是否覆盖 PRD §5 全量映射表所有页面(P0+P1+P2 · 不漏 / 不重)
- 每个页面 6 项字段齐全:① 标题+URL ② 实现优先级 ③ ASCII 布局 ④ Element Plus 组件 ⑤ 字段行为 ⑥ 跳转关系
- ASCII 布局是否真实存在(不是"略" / "见示意图"等占位)
- Element Plus 组件命名规范(`el-form` / `el-table` / `el-button` / `el-pagination` · 见 CLAUDE.md §三·五)
- 跳转关系闭合:每个页面知道"从哪来"+ "去哪里" + 异常分支(登录失败 / 取消 / 提交错误)
- 占位行 `## 6. 页面原型描述(由 /page-prototyper 追加)` 已被替换(不应保留)

### 2. 页面 ↔ PRD 字段对账(命中"§6 表单含 phone 但 PRD 没收集"问题)

逐页面检查:
- **参考集**(PRD 全文出现的业务字段):列出所有
- **被检集**(§6 各页面表单/列表/详情中提到的字段):列出所有
- **差集 / 结论**:
  - §6 引用但 PRD 找不到出处的字段 → 🔴 高严重度 issue(下游 db-designer 漏字段)
  - 收集类字段(phone / email / 头像 等)必须在 PRD 注册或资料维护流程中有收集环节,否则 → issue

### 3. 按钮 ↔ API 对账(命中"列表页画详情按钮但 PRD 没 GET /{id}"问题)

扫 §6 所有可交互元素("详情" / "删除" / "编辑" / "审核" / "导出" / "评价" / "分配" 等按钮 + 链接 + 图标点击):
- **参考集**(PRD §3 各功能「API 形态」字段中已声明的全部 API):列出
- **被检集**(§6 所有按钮):列出
- **双向对账**:
  - **正向**:UI 按钮 → 必有对应 API · 找不到 → 🔴 高严重度 issue(实现阶段 404 或超出 PRD)
  - **反向**:PRD 已声明的 API → 必有 UI 入口 · 找不到 → 🟡 中严重度 issue(死代码)
- **实现优先级一致**:UI 按钮所属页面的实现优先级 ≤ 对应 API 的实现优先级(P0 页面调用了 P1 接口 → 🔴 高严重度 issue)

### 4. 页面 ↔ §3 路由对账(命中"§6 URL 与 §3 路由表不一致"问题)

逐页面检查:
- §6 每个页面的 URL 是否在 §3 路由表中存在(URL 完全匹配 · 路径参数模式一致)
- §6 每个页面的「实现优先级」标签是否跟 §3 路由表同 URL 行的「实现优先级」列一致
- §6 每个页面的「跳转关系」中提到的目标 URL,目标页是否在 §3 / §6 中存在(不存在 → "断链" issue)
- §6 各页面"从哪来"反推出的来源页 → 该来源页的跳转关系中是否有跳到当前页(否则 § "孤儿页":没有任何页能跳到它 → issue)

### 5. 全局 UI 一致性(命中"AppLayout / 角色菜单未定义"问题)

- §6 中是否定义了**全局组件**(AppLayout / 顶栏 / 侧栏 / 面包屑 / 全局加载态 / 全局错误提示)· 不定义 → vue-page-coder 各页面各自实现导致风格混乱
- 是否定义了**角色菜单规则**(不同角色看到的菜单项不同 · 哪些菜单项随角色变化)
- 登录页 / 注册页等"特殊布局页"是否说明了"不套用 AppLayout"
- 全局组件本身是否作为独立子节列在 §6 中(如 `### 全局布局组件 AppLayout`)

## 设计要点

- **双模型策略**:写 §6 用一个模型,审 §6 用另一个,互相挑刺质量更高(对齐 R-01/R-02/R-03/R-04)
- **跨文档对账是核心**:维度 2/3/4 都强制输出"参考集 / 被检集 / 差集"三段式,不允许只给"无 issue"结论 —— 这是漏检根因专项防御
- **HTML 注释 + Git diff = 改前/改后证据**(05 验收要求 ≥5 处)
- **审核报告自动落盘**:学生不需要手动整理对话记录
- **「审核 ↔ 应用修复」二段循环**:本命令(R-02b)插 issue 注释 → page-prototyper §二 in-place 改为"已修复" → 双方协议:**同一注释格式 + 编号一致 + 生命周期闭环**
- **跟 R-02 互补 · 不重叠**:R-02 审 §1-§5(架构/路由/选型),R-02b 审 §6(页面原型)+ §6 跨文档对账。两者编号独立,各自插各自的注释
- **下游污染防护**:R-02b issue 必须在 Phase 2 db-designer 之前修复 —— §6 的字段/按钮/URL 直接影响 Phase 2-5 全部产出
