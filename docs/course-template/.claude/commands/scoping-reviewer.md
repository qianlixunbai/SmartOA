### 审核维度(5 维度 · 每维度有具体子项)

#### 1. 功能完整性

- **领域必备功能**:基于 § 一题名 + 一句话背景,该领域**必备**功能是否齐全?(用 AI 领域常识比对 · 例:电商必含订单/支付 · 物业必含业主/缴费 · 教务必含选课/成绩 · 旅行规划必含行程/费用 · 二手交易必含商品/交易状态)
- **关键业务流齐全**:有缴费就该有账单生成,有提交就该有处理/审核,有发布就该有查看 —— **不能有"半截功能"**(只设计起点没设计终点)
- **角色覆盖完整**:§ 一 JWT 角色字段是否覆盖该领域的**所有典型用户**?(例:学生请假系统 § 一只写"单一用户角色"但实际需要"学生 + 班主任 + 教务"三角色才能闭环)

#### 2. 功能合理性(教学可行性 + 跟约束冲突)

- **教学可行性**:P0 是否有**超出教学范围**的功能?(例:要求 OCR 识别身份证 / AI 智能推荐 / 实时音视频通话 / 微信支付集成 —— 这些不在 CLAUDE.md §一·一 技术栈支持范围,且 3 周做不完)
- **复杂度合理**:P0 是否有"看似简单实则复杂"功能?(例:多级权限审批工作流 / 复杂状态机 / 实时消息推送)
- **跟教学约束冲突**:文件上传是否要求云存储 OSS?(教学统一**本地 uploads/** · 见 CLAUDE.md §二·五)· 是否要求微信/支付宝集成?(教学避免外部账号依赖)
- **跟技术栈冲突**:是否隐含要求超出 SpringBoot 3 + Vue 3 + MySQL 的能力?(例:要求 NoSQL 必备 / 要求 ElasticSearch 全文检索 P0 必做)

#### 3. 角色与权限清晰度

- **角色定义齐全**:§ 一 JWT 角色字段是否模糊(如"单一用户角色")?在 § 三 P0 功能里隐含的角色是否一致(例:P0 描述"管理员审批投诉"但 § 一 JWT 角色没定义"管理员")
- **角色权限明确**:每个角色能做什么 / 不能做什么是否说清楚?(例:"老师"角色 · 是只能查看学生信息 / 还是能修改 / 是否能审批?)
- **跨角色业务关系**:角色之间的依赖关系是否合理(例:业主-房屋-物业费 是 1:N:N 关系 · 标定卡 § 一核心实体应反映)
- **数据隔离**:多角色场景下,数据是否应该隔离(用户 A 只能看自己数据)?标定卡是否暗示这一点(P0 vs P1 起隔离的合理性)

#### 4. 业务闭环性

- **主业务流端到端**:取领域**最重要的 1-2 个业务流**(如缴费/请假/购物),从起点到终点每一步是否都有对应功能?(例:缴费流 = 账单生成 → 通知用户 → 用户支付 → 财务确认 → 状态更新 · 不能漏中间环节)
- **数据流闭环**:对每个核心实体,**创建 + 查询 + 更新 + 删除** 4 类操作是否齐全?(P0 不需要全 CRUD · 但起码要有"创建 + 查询" · 然后看业务需要补 U/D)
- **上下游依赖完整**:P0 功能是否依赖 P1/P2 功能(错位)?(例:P0 "在线缴费"依赖 "账单生成" · 但"账单生成"在 P1 → 学生 P0 跑不通)
- **无半截功能**:有没有"设计了但下游缺失"?(例:设计了"投诉提交"但没"投诉处理"/ 设计了"商品发布"但没"商品下架")

#### 5. P0/P1/P2 划分合理性 ⭐ 重点

- **P0 必做合理性**:被标 P0 的功能是否真的"3 周内必须做完才立项"?有没有功能被**错升 P0**(其实 P1 也行 · 不影响立项验收)增加学生压力?
- **P1/P2 降级合理性**:
  - P1 是否真的"P0 跑通后的常见扩展"?
  - P2 是否真"加分项"?
  - 有没有 **P0 的关键依赖被错降 P1/P2**(导致 P0 跑不通 · 高严重度 issue)?
- **跨档依赖**:P0 功能是否依赖 P1/P2 功能(错位)?如发现 → 必须**升级**该 P1/P2 为 P0 · 或**降级**该 P0 为 P1/P2 一致档位
- **数量分布合理**:跟 § 二量化锚点对齐(P0 约 X 接口/Y 表/Z 页面 · 教学规律 P0 8-12 接口/4-5 表/4-6 页面 · P1 5-8 接口扩展 · P2 3-5 接口加分)· 偏离过大需注明
- **跟教学验收档位对齐**:P0 = 60 分基础 · P1 = 70-80 分 · P2 = 85+ 加分(母版 § 七 已说明 · 学生本地 docs/00-选题标定.md 不含此段)· 划分应让"分档拉开梯度"(不能 P0 太难 / 也不能 P1/P2 跟 P0 太接近显得多余)

## §二 应用修复模式(R-00 issue 处理 · 二级用法 · 一身两任 · 跟 prompt-evolver 同模式)

### 触发场景

`/scoping-reviewer` §一 完成审核后,docs/00-选题标定.md 中已有 `<!-- R-00-issue-编号: 严重度 - 描述 -->` HTML 注释。此时再次调用本命令进入"应用修复"模式 —— **AI 直接改标定卡**(包括 P0/P1/P2 划分 + 补缺漏 + 改不合理 · 不再是"提醒清单等学生手动改")。

> ⚠️ **本模式权限**:**AI 直接改 docs/00-选题标定.md 内容**(包括 P0/P1/P2 划分调整 / 补缺漏功能 / 改不合理设计 / 角色补全)· 学生只需 review diff 后 Accept · 调整理由自动落到 R-00 报告"修改决策记录"段作为答辩素材。
>
> ⚠️ **学生权限边界**:学生本地副本 docs/00-选题标定.md 允许调整(参 03-选题库 母版 § 七 P0 调整审计要求)· 调整记录答辩时学生说清楚理由即可 · 评委会认可这是企业需求二次评审。

### 输入

- **必读**:`docs/00-选题标定.md`(reviewer 已插入注释的版本)
- **必读**:`docs/对话记录/Phase0-R00-scoping-review-<日期>.md`(reviewer 报告 · 含每条 issue 的修复建议 + 修改决策记录占位段待回填)
- **可能必读**(按 issue 类别决定):
  - 根目录 `CLAUDE.md` §一·一(技术栈基线)+ §一·二(安全规范) —— 当 R-00 issue 落在**维度 2 功能合理性**(教学可行性/技术栈冲突 · 修复时需对照 §一·一 技术栈基线)
  - `03-选题库-学生标定卡-母版.md` —— 当 R-00 issue 涉及"是否跟教师母版一致" · 修复参考母版同题段
- 用户调用形式:`/scoping-reviewer 应用修复` 或 `/scoping-reviewer 请扫描 R-00 注释逐条修复`

### 输出指令(Claude Code 必须 3 项都做)

1. 扫描 docs/00-选题标定.md 中所有 `<!-- R-00-issue-... -->` 注释
2. 对每条注释:
   - **直接修改**对应章节内容(基于 reviewer 报告的修复建议):
     - 维度 1 功能完整性 issue → **补缺漏功能** 到对应 §三/§四/§五 清单
     - 维度 2 功能合理性 issue → **改不合理设计**(降级超教学范围功能 / 改本地存储 / 简化复杂度)
     - 维度 3 角色 issue → **补全角色定义** 到 § 一 JWT 角色字段 + § 三 P0 功能描述
     - 维度 4 业务闭环 issue → **补半截功能下游** 或 **删冗余功能**
     - 维度 5 P0/P1/P2 划分 issue → **调整划分**(P0↔P1 升降级 / P1↔P2 升降级 / 跨档依赖修正)
   - 把注释改写为 `<!-- R-00-issue-编号: 已修复 - 一句话修复说明 -->`
3. **回填 R-00 报告 "修改决策记录" 段**:
   - 把每个 issue 的"原标定/调整后/理由"按报告模板格式填入 review.md 的 "## 修改决策记录(答辩素材)" 段
   - 内容来自 issue 的"修复建议" + 实际修改后的标定卡内容
4. 输出 diff(显示每个 issue 的改前/改后对比 + review.md 修改决策记录段回填)

### 输出自检 checklist(应用修复模式)

- [ ] 标定卡中所有 R-00 注释都已标记"已修复"(没遗漏)
- [ ] 修复内容覆盖 reviewer 报告的 issue 要点
- [ ] § 一基本信息中的题号 / 题名 / 一句话背景 **完全未被触动**(那是教师标定 · 学生不动)
- [ ] § 二量化锚点(P0/P1/P2 表数/接口/页面/工时)**若调整需明示理由**(在 review.md 修改决策记录段)
- [ ] § 三 P0 / § 四 P1 / § 五 P2 内容**按 R-00 issue 调整**(可补缺漏 / 可调划分 / 可改不合理)
- [ ] 标定卡 5 大段(§ 一~§ 五)结构未破坏(不增不减大节)
- [ ] R-00 报告 "修改决策记录" 段已完整回填(每个 issue 1 段 · 答辩素材就绪)
- [ ] 输出 diff 含改前/改后对比(便于学生 review)

- **跟下游 R-XX 边界**:
  - **R-00 审 P0/P1/P2 划分合理性** + 标定卡缺漏 + 角色完整性
  - **R-01 审 PRD 是否完整反映已审过的标定卡** + PRD 各功能字段是否清晰可验证(**不重复审 P0/P1/P2 划分** · 那是 R-00 的事)
- **HTML 注释 + Git diff = 改前/改后证据**(05 验收要求 ≥5 处 · R-00 也算 1 处)
- **审核报告自动落盘**(对齐 R-01/R-02/R-03/R-04 模式):学生不需要手动整理对话记录
