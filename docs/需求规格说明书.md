# 需求规格说明书

## SmartOA — 简易 OA 审批流管理系统

---

## 1. 引言

### 1.1 编写目的

本文档旨在详细描述 SmartOA 审批流管理系统的功能需求和非功能需求，作为开发、测试和验收的依据。

### 1.2 项目背景

企业内部日常审批（请假、报销、加班等）传统上依赖纸质单据或碎片化的即时通讯工具，存在效率低、易丢失、难追溯等问题。SmartOA 提供一套轻量级的线上审批解决方案。

### 1.3 适用范围

- 开发团队：理解需求细节，指导编码实现
- 测试团队：编写测试用例的依据
- 用户/评审方：验收功能是否完整

### 1.4 术语定义

| 术语 | 说明 |
|---|---|
| 审批模板 | 预定义的审批规则，包含表单字段和审批流程节点 |
| 审批节点 | 审批流程中的一个步骤，定义了由谁来审批 |
| 审批流转 | 申请从提交到最终通过/驳回的完整路径 |
| 直属领导 | 组织架构中员工的直接上级 |
| 部门总监 | 部门负责人 |

## 2. 系统总体描述

### 2.1 产品视角

SmartOA 是一个基于 B/S 架构的 Web 应用系统，采用前后端分离架构。前端使用 Vue 3 + Element Plus 构建 SPA，后端使用 Spring Boot 提供 RESTful API，数据存储在 MySQL 8 中。

### 2.2 用户角色

| 角色 | 职责 | 权限级别 |
|---|---|---|
| EMPLOYEE（普通员工） | 提交申请，查看个人申请历史，撤回自己的申请 | 基础 |
| MANAGER（管理员/审批人） | 审批待办、管理模板、导出数据、查看报表 | 高级 |

### 2.3 运行环境

- 服务端：JDK 21 + Spring Boot 4.0.6
- 数据库：MySQL 8.0+
- 客户端：现代浏览器（Chrome / Firefox / Edge）
- 网络：HTTP/HTTPS

## 3. 功能性需求

### 3.1 用户认证模块 (AUTH)

#### 3.1.1 登录

| 需求编号 | AUTH-001 |
|---|---|
| 描述 | 用户通过用户名和密码登录系统 |
| 输入 | 用户名（username）、密码（password） |
| 处理 | 查询用户表验证凭据，生成 JWT Token |
| 输出 | Token + 用户基本信息（不含密码） |
| 异常 | 用户名或密码错误时返回失败提示 |
| Token 有效期 | 24 小时（86400000ms） |

#### 3.1.2 认证拦截

| 需求编号 | AUTH-002 |
|---|---|
| 描述 | 除登录接口外，所有 /api/* 接口需 JWT 认证 |
| 处理 | JwtFilter 拦截请求，验证 Authorization 头中的 Bearer Token |
| 异常 | 无 Token / Token 无效 / Token 过期 / 用户不存在 → 401 |

#### 3.1.3 获取当前用户

| 需求编号 | AUTH-003 |
|---|---|
| 描述 | 客户端可根据 Token 获取当前登录用户信息 |
| 输出 | 用户 ID、用户名、姓名、角色、部门、组织层级 ID |

#### 3.1.4 用户列表

| 需求编号 | AUTH-004 |
|---|---|
| 描述 | 前端获取所有用户列表（ID → 姓名映射用） |
| 输出 | User[] 数组，password 字段被 @JsonIgnore 排除 |

### 3.2 审批模板管理模块 (TPL)

#### 3.2.1 模板列表

| 需求编号 | TPL-001 |
|---|---|
| 描述 | 管理员查看所有审批模板 |
| 权限 | MANAGER |

#### 3.2.2 模板 CRUD

| 需求编号 | TPL-002 ~ TPL-005 |
|---|---|
| 描述 | 创建、查看、编辑、删除审批模板 |
| 模板字段 | name（模板名称）、description（描述）、enabled（是否启用） |
| 删除规则 | 删除模板时级联删除关联的审批节点和表单字段 |

#### 3.2.3 审批节点管理

| 需求编号 | TPL-006 |
|---|---|
| 描述 | 管理员为模板配置审批流程节点 |
| 节点字段 | nodeName（节点名称）、sortOrder（顺序）、approverType（审批人类型）、approverId（指定审批人） |
| 处理 | 保存时先删旧节点，再批量插入新节点，自动分配 sortOrder |
| 约束 | 同一模板下 sortOrder 唯一 |

#### 3.2.4 审批人类型

| 类型代码 | 解析规则 |
|---|---|
| DIRECT_LEADER | 查找申请人的 directLeaderId |
| DEPARTMENT_HEAD | 查找申请人的 departmentHeadId |
| SPECIFIC_USER | 使用节点配置的 approverId |

#### 3.2.5 表单字段管理

| 需求编号 | TPL-007 |
|---|---|
| 描述 | 管理员查看模板的表单字段定义 |
| 字段类型 | TEXT / DATE / SELECT / TEXTAREA / NUMBER |
| SELECT 选项 | JSON 数组格式存储 |

### 3.3 请假申请模块 (LEV)

#### 3.3.1 提交申请

| 需求编号 | LEV-001 |
|---|---|
| 描述 | 员工提交请假申请，系统自动启动审批流程 |
| 输入 | templateId、leaveType、startDate、endDate、reason |
| 处理 | 1. 验证用户存在 2. 创建 LeaveRequest 3. 查模板的首个审批节点 4. 解析审批人 5. 设置 currentNodeId + currentApproverId |
| 前置条件 | 申请人已分配直属领导 |
| 状态 | PENDING |

#### 3.3.2 审批操作

| 需求编号 | LEV-002 |
|---|---|
| 描述 | 审批人对申请进行通过或驳回操作 |
| 输入 | requestId、action（APPROVE/REJECT）、comment |
| 校验 | 1. 申请存在 2. 状态为 PENDING 3. 操作人是 currentApproverId |
| 通过处理 | 记录审批 → 找下一节点 → 有则流转，无则标记 APPROVED |
| 驳回处理 | 记录审批 → 标记 REJECTED → 流程终止 |

#### 3.3.3 撤回申请

| 需求编号 | LEV-003 |
|---|---|
| 描述 | 申请人在审批完成前撤回申请 |
| 权限 | 仅申请人 |
| 前置条件 | 状态为 PENDING |
| 处理 | 状态 → WITHDRAWN，currentApproverId → null，currentNodeId → null，写审批记录 |

#### 3.3.4 转派审批

| 需求编号 | LEV-004 |
|---|---|
| 描述 | 当前审批人将审批任务转给其他人 |
| 输入 | toUserId |
| 权限 | 当前审批人 |
| 处理 | 验证目标用户 → 改 currentApproverId → 写转派记录（含 comment "转派给 xxx"） |
| 状态 | 保持 PENDING，不改变审批节点 |

#### 3.3.5 我的申请列表

| 需求编号 | LEV-005 |
|---|---|
| 描述 | 查看当前用户提交的所有申请 |
| 排序 | 按创建时间倒序 |

#### 3.3.6 待审批列表

| 需求编号 | LEV-006 |
|---|---|
| 描述 | 查看当前用户需要审批的 PENDING 状态申请 |
| 查询条件 | currentApproverId = 当前用户 AND status = PENDING |

#### 3.3.7 已处理列表

| 需求编号 | LEV-007 |
|---|---|
| 描述 | 查看当前用户处理过的申请 |
| 查询条件 | currentApproverId = 当前用户 AND status != PENDING |

#### 3.3.8 申请详情

| 需求编号 | LEV-008 |
|---|---|
| 描述 | 查看申请完整信息，包含申请人信息、表单数据、状态 |

#### 3.3.9 审批记录

| 需求编号 | LEV-009 |
|---|---|
| 描述 | 查看申请的所有审批操作记录 |
| 排序 | 按创建时间升序（时间线展示） |

## 5. 非功能性需求

### 5.1 性能

| 指标 | 要求 |
|---|---|
| 接口响应时间 | ≤ 500ms（简单查询） |
| 页面加载时间 | ≤ 2s |
| 并发用户 | 50（演示/学习场景） |

### 5.2 安全性

| 要求 | 实现 |
|---|---|
| 认证 | JWT Bearer Token |
| 密码保护 | 不返回给前端（@JsonIgnore） |
| 越权防护 | 操作前校验身份（撤回=申请人，审批=当前审批人） |
| CORS | 仅允许前端域名跨域 |
| 密码加密 | 当前演示用明文，生产应替换为 BCrypt |

### 5.3 可用性

- 表单提供实时校验和错误提示
- 操作有成功/失败反馈
- 按钮状态与权限联动（无权限时隐藏）
- 审批进度条直观展示流程位置

### 5.4 可维护性

- 后端：分层架构（Controller → Service → Mapper），职责清晰
- 前端：Pinia 状态管理 + API 层独立
- 数据库：迁移脚本版本化管理

### 5.5 可扩展性

- 审批节点类型可新增（修改 approverType 枚举 + resolveApprover 方法）
- 模板字段类型可扩展
- 可增加其他业务类型（报销、加班等）复用审批引擎
