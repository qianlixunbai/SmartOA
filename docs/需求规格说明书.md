# 需求规格说明书

## SmartOA — 简易 OA 审批流管理系统

---

## 1. 引言

### 1.1 编写目的

本文档旨在详细描述 SmartOA 审批流管理系统的功能需求和非功能需求，作为开发、测试和验收的依据。

### 1.2 项目背景

企业内部日常审批（请假、报销、加班等）传统上依赖纸质单据或碎片化的即时通讯工具，存在效率低、易丢失、难追溯等问题。SmartOA 提供一套轻量级的线上审批解决方案。

### 1.3 适用范围

- 开发团队：理解需求细节，指导编码实现
- 测试团队：编写测试用例的依据
- 用户/评审方：验收功能是否完整

### 1.4 术语定义

| 术语 | 说明 |
|---|---|
| 审批模板 | 预定义的审批规则，包含表单字段和审批流程节点 |
| 审批节点 | 审批流程中的一个步骤，定义了由谁来审批 |
| 审批流转 | 申请从提交到最终通过/驳回的完整路径 |
| 直属领导 | 组织架构中员工的直接上级 |
| 部门总监 | 部门负责人 |
| 会签 (COUNTER_SIGN) | 节点上所有审批人都同意才进入下一步 |
| 或签 (OR_SIGN) | 任一审批人同意即进入下一步 |
| 条件分支 | 根据请假类型/天数等条件动态跳过或触发审批节点 |
| 超时自动升级 | 审批超时后自动执行指定动作（转派/通过/驳回） |

## 2. 系统总体描述

### 2.1 产品视角

SmartOA 是一个基于 B/S 架构的 Web 应用系统，采用前后端分离架构。前端使用 Vue 3 + Element Plus 构建 SPA，后端使用 Spring Boot + MyBatis-Plus 提供 RESTful API，数据存储在 MySQL 8 中。

### 2.2 用户角色

| 角色 | 职责 | 权限级别 |
|---|---|---|
| EMPLOYEE（普通员工） | 提交申请，查看个人申请历史，撤回自己的申请 | 基础 |
| MANAGER（管理员/审批人） | 审批待办、管理模板、导出数据、查看报表 | 高级 |

### 2.3 运行环境

- 服务端：JDK 21 + Spring Boot 4.0.6
- 数据库：MySQL 8.0+
- 客户端：现代浏览器（Chrome / Firefox / Edge）
- 网络：HTTP/HTTPS

## 3. 功能性需求

### 3.1 用户认证模块 (AUTH)

#### 3.1.1 登录

| 需求编号 | AUTH-001 |
|---|---|
| 描述 | 用户通过用户名和密码登录系统 |
| 输入 | 用户名（username）、密码（password） |
| 处理 | 查询用户表验证 BCrypt 密码，生成 JWT Token |
| 输出 | `Result<Map>` 含 Token + 用户基本信息（不含密码） |
| 异常 | 用户名或密码错误时返回失败提示 |
| Token 有效期 | 24 小时（86400000ms） |

#### 3.1.2 认证拦截

| 需求编号 | AUTH-002 |
|---|---|
| 描述 | 除登录接口外，所有 /api/* 接口需 JWT 认证 |
| 处理 | JwtFilter 拦截请求，验证 Authorization 头中的 Bearer Token |
| 异常 | 无 Token / Token 无效 / Token 过期 / 用户不存在 → 401 |

#### 3.1.3 获取当前用户

| 需求编号 | AUTH-003 |
|---|---|
| 描述 | 客户端可根据 Token 获取当前登录用户信息 |
| 输出 | 用户 ID、用户名、姓名、角色、部门、组织层级 ID |

#### 3.1.4 用户列表

| 需求编号 | AUTH-004 |
|---|---|
| 描述 | 前端获取所有用户列表（用于选择审批人） |
| 输出 | User[] 数组，password 字段被 @JsonIgnore 排除 |

### 3.2 审批模板管理模块 (TPL)

#### 3.2.1 模板列表

| 需求编号 | TPL-001 |
|---|---|
| 描述 | 管理员查看所有审批模板 |
| 权限 | MANAGER |

#### 3.2.2 模板 CRUD

| 需求编号 | TPL-002 ~ TPL-005 |
|---|---|
| 描述 | 创建、查看、编辑、删除审批模板 |
| 模板字段 | name（模板名称）、description（描述）、enabled（是否启用） |
| 删除规则 | 删除模板时级联删除关联的审批节点和表单字段 |

#### 3.2.3 审批节点管理

| 需求编号 | TPL-006 |
|---|---|
| 描述 | 管理员为模板配置审批流程节点，支持可视化拖拽排序 |
| 节点字段 | nodeName、sortOrder、approverType、approverId、signType、approverIds、conditionExpression、timeoutHours、timeoutAction、escalateToUserId |
| 处理 | 保存时先删旧节点（清理引用），再批量插入新节点，自动分配 sortOrder |

#### 3.2.4 审批人类型

| 类型代码 | 解析规则 |
|---|---|
| DIRECT_LEADER | 查找申请人的 directLeaderId |
| DEPARTMENT_HEAD | 查找申请人的 departmentHeadId |
| SPECIFIC_USER | 使用节点配置的 approverId（单人）/ approverIds（并行） |

#### 3.2.5 签批模式（P2 新增）

| 需求编号 | TPL-008 |
|---|---|
| 描述 | 节点支持三种签批模式 |
| 类型 | SINGLE（单人审批）、COUNTER_SIGN（会签，全部同意才推进）、OR_SIGN（或签，任一人同意即推进） |
| 默认 | SINGLE |

#### 3.2.6 条件分支（P2 新增）

| 需求编号 | TPL-009 |
|---|---|
| 描述 | 审批节点支持 SpEL 条件表达式，满足条件才触发 |
| 变量 | leaveType（请假类型）、days（请假天数）、startDate、endDate |
| 示例 | `days > 3` → 超过3天走此节点；`leaveType == '病假'` → 病假走此节点 |

#### 3.2.7 超时配置（P2 新增）

| 需求编号 | TPL-010 |
|---|---|
| 描述 | 节点可配置超时自动处理策略 |
| 参数 | timeoutHours（超时小时数）、timeoutAction（ESCALATE/AUTO_APPROVE/AUTO_REJECT）、escalateToUserId（转派目标） |

#### 3.2.8 表单字段管理

| 需求编号 | TPL-007 |
|---|---|
| 描述 | 管理员查看模板的表单字段定义 |
| 字段类型 | TEXT / DATE / SELECT / TEXTAREA / NUMBER |
| SELECT 选项 | JSON 数组格式存储 |

### 3.3 请假申请模块 (LEV)

#### 3.3.1 提交申请

| 需求编号 | LEV-001 |
|---|---|
| 描述 | 员工提交请假申请，系统自动启动审批流程 |
| 输入 | templateId、leaveType、startDate、endDate、reason |
| 处理 | 1. 验证用户存在 2. 创建 LeaveRequest 3. 推进到第一个满足条件的审批节点 4. 根据节点模式设置 currentApproverId 或创建 approval_task |
| 前置条件 | 申请人已分配直属领导 |
| 状态 | PENDING |

#### 3.3.2 审批操作

| 需求编号 | LEV-002 |
|---|---|
| 描述 | 审批人对申请进行通过或驳回操作 |
| 输入 | requestId、action（APPROVE/REJECT）、comment |
| 校验 | 1. 申请存在 2. 状态为 PENDING 3. 操作人匹配（SINGLE: currentApproverId；并行: approval_task 中有 PENDING 任务） |
| 通过处理 | 单人→推进；或签→跳过其余+推进；会签→如全部完成则推进 |
| 驳回处理 | 记录审批 → 标记 REJECTED → 跳过并行任务 → 流程终止 |

#### 3.3.3 撤回申请

| 需求编号 | LEV-003 |
|---|---|
| 描述 | 申请人在审批完成前撤回申请 |
| 权限 | 仅申请人 |
| 前置条件 | 状态为 PENDING |
| 处理 | 状态 → WITHDRAWN，跳过并行任务，currentApproverId/currentNodeId/timeoutTime → null |

#### 3.3.4 转派审批

| 需求编号 | LEV-004 |
|---|---|
| 描述 | 当前审批人将审批任务转给其他人（仅 SINGLE 模式） |
| 输入 | toUserId |
| 权限 | 当前审批人 |
| 处理 | 验证目标用户 → 改 currentApproverId → 写转派记录 |
| 约束 | 并行审批节点不支持转派 |

#### 3.3.5 并行审批任务查询（P2 新增）

| 需求编号 | LEV-010 |
|---|---|
| 描述 | 查询请假单当前节点的 PENDING 审批任务列表 |
| 输出 | ApprovalTask[] 含 approverId、status |

#### 3.3.6 滞留修复（P2 新增）

| 需求编号 | LEV-011 |
|---|---|
| 描述 | 修复 currentApproverId 为 null 且无关 PENDING 任务的滞留申请 |
| 触发 | 手动调用 POST /api/leave/repair 或自动调度 |

#### 3.3.7 我的申请列表

| 需求编号 | LEV-005 |
|---|---|
| 描述 | 查看当前用户提交的所有申请 |

#### 3.3.8 待审批列表

| 需求编号 | LEV-006 |
|---|---|
| 描述 | 查看当前用户需要审批的 PENDING 状态申请（含 SINGLE + 并行任务） |
| 查询条件 | currentApproverId = 当前用户 OR approval_task 中有 PENDING 任务 |

#### 3.3.9 已处理列表

| 需求编号 | LEV-007 |
|---|---|
| 描述 | 查看当前用户处理过的申请 |

#### 3.3.10 申请详情

| 需求编号 | LEV-008 |
|---|---|
| 描述 | 查看申请完整信息，含 timeoutTime、审批节点配置 |

#### 3.3.11 审批记录

| 需求编号 | LEV-009 |
|---|---|
| 描述 | 查看申请的所有审批操作记录（含 TIMEOUT_* 系统自动操作） |

### 3.4 超时自动升级模块 (TMO) — P2 新增

#### 3.4.1 超时检查

| 需求编号 | TMO-001 |
|---|---|
| 描述 | 系统每 5 分钟自动扫描 PENDING 申请，检查 timeout_time 是否过期 |
| 触发 | @Scheduled(fixedRate = 300000) |
| 处理 | 对每个超时申请执行节点配置的 timeoutAction |

#### 3.4.2 超时动作

| 动作 | 说明 |
|---|---|
| ESCALATE | 转派给 escalateToUserId 或跳过当前节点进入下一步 |
| AUTO_APPROVE | 自动通过当前节点，跳过剩余审批人，进入下一步 |
| AUTO_REJECT | 自动驳回申请，终止流程 |

#### 3.4.3 超时记录

| 需求编号 | TMO-003 |
|---|---|
| 描述 | 超时处理写入 ApprovalRecord（approverId=0, action=TIMEOUT_*），日志可追溯 |

## 5. 非功能性需求

### 5.1 性能

| 指标 | 要求 |
|---|---|
| 接口响应时间 | ≤ 500ms（简单查询） |
| 页面加载时间 | ≤ 2s |
| 定时任务间隔 | 5 分钟（超时检查） |

### 5.2 安全性

| 要求 | 实现 |
|---|---|
| 认证 | JWT Bearer Token（jjwt 0.13.0） |
| 密码保护 | BCrypt 加密（spring-security-crypto），不返回给前端 |
| 越权防护 | 操作前校验身份（撤回=申请人，审批=当前审批人/并行任务） |
| CORS | 仅允许前端域名跨域 |

### 5.3 可用性

- 表单提供实时校验和错误提示
- 操作有成功/失败反馈（ElMessage）
- 按钮状态与权限联动（无权限时隐藏）
- 审批进度条直观展示流程位置
- 超时警告栏提示截止时间

### 5.4 可维护性

- 后端：分层架构（Controller → Service → Mapper），统一响应 `Result<T>`
- 前端：Pinia 状态管理 + API 层独立
- 数据库：迁移脚本版本化管理（mysql-p0 ~ mysql-p5）
- 全局异常处理：BusinessException + @RestControllerAdvice
